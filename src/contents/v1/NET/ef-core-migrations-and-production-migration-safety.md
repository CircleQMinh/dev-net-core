## Overview

Entity Framework Core migrations are a way to evolve a relational database schema as the application model changes. Instead of manually updating the database every time an entity, relationship, index, constraint, or column changes, EF Core can compare the current model with the previous model snapshot and generate migration files that describe the schema change.

This topic matters because database changes are among the riskiest parts of software delivery. A bad migration can drop data, lock large tables, break old application versions, fail halfway through deployment, or make rollback difficult. In production systems, migrations are not just a developer convenience. They are part of release management, operational safety, data protection, and system design.

In real projects, EF Core migrations are commonly used for:

- Creating and evolving tables, columns, indexes, relationships, and constraints.
- Keeping development, staging, and production schemas aligned.
- Reviewing generated SQL before deployment.
- Applying schema changes through CI/CD pipelines.
- Coordinating application deployments with database deployments.
- Preparing rollback and recovery plans.
- Managing data migrations when schema changes require moving or transforming data.

Interviewers ask about this topic because it reveals whether a developer understands both EF Core mechanics and production database safety. A strong answer should show that you can generate migrations, inspect generated files, produce SQL scripts, review risk, test migrations, plan rollback, and avoid unsafe deployment habits such as blindly running schema changes from application startup.

## Core Concepts

### What EF Core migrations are

An EF Core migration is a generated code file that describes how to change the database schema from one version to another.

A typical migration contains:

- An `Up` method that applies the change.
- A `Down` method that reverses the change.
- Metadata that identifies the migration.
- A model snapshot update that represents the latest EF Core model.

Example migration:

```csharp
using Microsoft.EntityFrameworkCore.Migrations;

#nullable disable

public partial class AddCustomerEmail : Migration
{
    protected override void Up(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.AddColumn<string>(
            name: "Email",
            table: "Customers",
            type: "nvarchar(320)",
            maxLength: 320,
            nullable: true);

        migrationBuilder.CreateIndex(
            name: "IX_Customers_Email",
            table: "Customers",
            column: "Email");
    }

    protected override void Down(MigrationBuilder migrationBuilder)
    {
        migrationBuilder.DropIndex(
            name: "IX_Customers_Email",
            table: "Customers");

        migrationBuilder.DropColumn(
            name: "Email",
            table: "Customers");
    }
}
```

The migration is source code. It should be reviewed like any other code change.

### Why migrations exist

Applications change over time. Entities gain new properties, indexes are added for performance, relationships change, columns become required, and old fields are removed. The database schema must evolve with the application model.

EF Core migrations help by:

- Recording incremental schema changes.
- Making schema changes repeatable.
- Allowing teams to apply the same changes across environments.
- Keeping changes in source control.
- Generating SQL scripts from migration code.
- Supporting rollback script generation.

Migrations do not remove the need for database review. EF Core can generate useful code, but the developer is still responsible for validating whether the generated migration is safe.

### Key terminology

| Term | Meaning |
|---|---|
| Migration | A versioned schema change generated by EF Core. |
| `Up` method | Applies the migration. |
| `Down` method | Reverses the migration. |
| Model snapshot | A C# file representing EF Core's understanding of the current model after the latest migration. |
| `__EFMigrationsHistory` | A database table used by EF Core to track which migrations have already been applied. |
| Idempotent script | A SQL script that can safely apply only migrations that are not already present in the target database. |
| Migration bundle | A single executable generated from EF Core migrations that can apply migrations without the full project source code. |
| Data migration | A migration that changes existing data, not just schema. |
| Rollback | Reverting a database to a previous migration or restoring from backup. |
| Expand-and-contract | A deployment pattern that makes database changes in backward-compatible stages. |
| Breaking schema change | A schema change that can break an existing application version. |

### How EF Core generates a migration

When a developer runs a command such as:

```bash
dotnet ef migrations add AddCustomerEmail
```

EF Core compares:

1. The current model built from the `DbContext`.
2. The previous model stored in the migration snapshot.

It then generates migration code that represents the difference.

Example workflow:

```bash
dotnet ef migrations add AddCustomerEmail
dotnet ef migrations script
dotnet ef database update
```

In Visual Studio Package Manager Console, the equivalent commands are commonly:

```powershell
Add-Migration AddCustomerEmail
Script-Migration
Update-Database
```

The migration name should describe the change, similar to a commit message.

Good names:

```text
AddCustomerEmail
CreateOrderAuditTable
AddIndexToInvoiceNumber
MakeProductSkuRequired
```

Poor names:

```text
Update1
FixStuff
NewMigration
Changes
```

### Migration files and model snapshot

When a migration is added, EF Core usually generates files similar to:

```text
Migrations/
  20260516101523_AddCustomerEmail.cs
  20260516101523_AddCustomerEmail.Designer.cs
  ApplicationDbContextModelSnapshot.cs
```

The main migration file contains the `Up` and `Down` methods.

The designer file contains migration metadata.

The snapshot file represents the current model state after all migrations.

The snapshot is important because EF Core uses it to determine what changed when creating the next migration. If the snapshot is corrupted or incorrectly merged, future migrations can become wrong.

In team environments, snapshot merge conflicts should be handled carefully. Do not blindly accept one side of a merge conflict. Build the project, add a test migration if needed, inspect the generated operations, and confirm that the final snapshot accurately represents the intended model.

### `__EFMigrationsHistory`

EF Core creates a table named `__EFMigrationsHistory` by default. This table records which migrations have been applied to a database.

A simplified example:

```sql
SELECT MigrationId, ProductVersion
FROM __EFMigrationsHistory
ORDER BY MigrationId;
```

Example result:

```text
20260510120000_InitialCreate
20260512143000_AddCustomerEmail
20260515100000_CreateOrderAuditTable
```

EF Core uses this table to know which migrations are pending.

Important points:

- The table tracks applied migrations, not every possible model state.
- Do not manually edit this table unless you fully understand the consequences.
- If the table is missing, incorrect, or out of sync, EF Core may try to apply the wrong migrations.
- Idempotent scripts use this table to skip already-applied migrations.

### Applying migrations during development

For local development, it is common to run:

```bash
dotnet ef database update
```

This applies pending migrations to the configured database.

You can update to a specific migration:

```bash
dotnet ef database update AddCustomerEmail
```

You can also roll back locally by updating to an earlier migration:

```bash
dotnet ef database update InitialCreate
```

For local development, this workflow is productive and usually acceptable.

For production, direct use of `dotnet ef database update` is usually not the safest choice because it applies changes directly without a separate SQL review step.

### SQL scripts for production deployment

A common production-safe approach is to generate SQL from EF Core migrations and review it before deployment.

Generate a script from an empty database to the latest migration:

```bash
dotnet ef migrations script
```

Generate a script from a specific migration to the latest migration:

```bash
dotnet ef migrations script AddCustomerEmail
```

Generate a script between two migrations:

```bash
dotnet ef migrations script AddCustomerEmail CreateOrderAuditTable
```

Generate an idempotent script:

```bash
dotnet ef migrations script --idempotent
```

A reviewed SQL script is often preferred in production because:

- The team can inspect exactly what will run.
- A DBA can review performance and locking impact.
- Dangerous operations such as column drops can be detected.
- The script can be archived with the release.
- The script can be tested against a restored production-like database.
- Deployment tooling can apply the script consistently.

### Idempotent SQL scripts

An idempotent script checks the migration history table and applies only missing migrations.

This is useful when:

- Different environments may be at different migration levels.
- Multiple tenant databases need the same migration package.
- The deployment pipeline does not know the exact current migration of every database.

Example command:

```bash
dotnet ef migrations script --idempotent -o migrations.sql
```

Idempotent scripts are convenient, but they should still be reviewed and tested. Custom SQL inside old migrations can sometimes cause provider-specific validation issues, especially if it references columns or tables that no longer exist. In important systems, some teams prefer generating targeted scripts from a known source migration to a known target migration.

### Migration bundles

EF Core migration bundles package migrations into an executable.

Example:

```bash
dotnet ef migrations bundle
```

A bundle can be useful because:

- It can be generated during CI.
- It can be executed during deployment.
- It does not require the project source code on the production server.
- It can avoid installing the full EF toolchain on the target machine.

A bundle still needs operational controls. It should be versioned, tested, reviewed, and executed with appropriate database permissions.

### Runtime migrations with `MigrateAsync`

EF Core allows applying migrations from application code:

```csharp
using var scope = app.Services.CreateScope();

var db = scope.ServiceProvider.GetRequiredService<ApplicationDbContext>();

await db.Database.MigrateAsync();
```

This can be useful for small internal tools, local demos, development environments, or simple single-instance deployments.

However, it is usually a poor production default because:

- The application needs schema modification permissions.
- Multiple application instances may attempt migration at startup.
- Users may access the database while schema changes are running.
- SQL is applied without a separate human review step.
- Failure during startup can take the whole application offline.
- Rollback control is weaker than a deliberate deployment script.

Newer EF Core versions provide more protection against concurrent migration execution, but that does not remove the broader production concerns around permissions, review, deployment timing, data safety, and rollback planning.

In interviews, a strong answer is usually:

> `MigrateAsync` can be useful in development or controlled environments, but for production I prefer reviewed scripts or migration bundles executed as part of the deployment process, with backups, testing, monitoring, and rollback planning.

### `EnsureCreated` vs migrations

`EnsureCreated` creates a database schema directly from the model without using migrations.

Example:

```csharp
await db.Database.EnsureCreatedAsync();
```

This is not the same as migrations.

Important differences:

| Feature | `EnsureCreated` | Migrations |
|---|---|---|
| Tracks schema history | No | Yes |
| Uses migration files | No | Yes |
| Good for production evolution | No | Yes |
| Good for quick tests or prototypes | Sometimes | Yes, but slower |
| Can evolve schema incrementally | No | Yes |

Do not mix `EnsureCreated` and `Migrate` for the same relational database lifecycle. If a database was created with `EnsureCreated`, migrations may not work correctly because the migration history table and migration metadata are missing.

### Reviewing generated migrations

Every generated migration should be reviewed.

Look for:

- Dropped columns or tables.
- Renames incorrectly detected as drop-and-add.
- Unintended nullable changes.
- New required columns without defaults.
- Indexes that may be expensive to create.
- Foreign keys that may fail due to existing data.
- Seed data updates caused by dynamic values.
- Large data updates inside a migration.
- Provider-specific SQL that may not run in all environments.
- Missing `Down` logic where rollback is expected.

Example problem: EF Core may detect a property rename as dropping one column and adding another.

Generated migration:

```csharp
migrationBuilder.DropColumn(
    name: "FullName",
    table: "Customers");

migrationBuilder.AddColumn<string>(
    name: "DisplayName",
    table: "Customers",
    type: "nvarchar(200)",
    maxLength: 200,
    nullable: false,
    defaultValue: "");
```

This could lose data.

A safer version may use a rename operation:

```csharp
migrationBuilder.RenameColumn(
    name: "FullName",
    table: "Customers",
    newName: "DisplayName");
```

Interview point: EF Core cannot always know developer intent. Developers must review migrations before applying them.

### Reviewing generated SQL scripts

Migration code is not enough. In production, the generated SQL should also be reviewed.

Generate SQL:

```bash
dotnet ef migrations script PreviousMigration NewMigration -o migration.sql
```

Review for:

- Table scans on large tables.
- Long-running locks.
- `ALTER TABLE` operations that rewrite data.
- Index creation impact.
- Foreign key validation impact.
- Data loss operations.
- Transaction behavior.
- Commands that cannot run inside a transaction.
- Provider-specific behavior.
- Whether the script matches the target environment's current schema.

A migration can look simple in C# but produce expensive SQL. Production review should focus on the actual SQL that will run.

### Rollback strategy

Rollback is not just the `Down` method. A real rollback strategy answers:

- Can the schema change be reversed safely?
- Is data lost during rollback?
- Does the old application version still work with the new schema?
- Is a backup available?
- Has the rollback script been tested?
- How long will rollback take?
- Who is authorized to execute it?
- What monitoring will confirm success or failure?

EF Core can generate a rollback script by reversing the migration range:

```bash
dotnet ef migrations script NewMigration PreviousMigration -o rollback.sql
```

However, rollback scripts have limits.

If a migration drops a column, rollback may recreate the column but cannot restore lost data unless the data was preserved somewhere.

Example dangerous migration:

```csharp
migrationBuilder.DropColumn(
    name: "LegacyCode",
    table: "Products");
```

The `Down` method can add the column back:

```csharp
migrationBuilder.AddColumn<string>(
    name: "LegacyCode",
    table: "Products",
    type: "nvarchar(50)",
    nullable: true);
```

But the original values are gone unless they were backed up or copied elsewhere.

### Backward-compatible migrations

A safe production migration should often be backward-compatible with both the old and new application versions.

This matters when using:

- Rolling deployments.
- Blue/green deployments.
- Canary releases.
- Multiple application instances.
- Microservices sharing a database.
- Long-running background workers.

A backward-compatible migration avoids breaking the currently running app.

Example safer approach for changing a required column:

1. Add the new column as nullable.
2. Deploy application code that writes both old and new columns.
3. Backfill existing rows.
4. Deploy code that reads from the new column.
5. Make the new column required.
6. Remove the old column in a later release.

This is often called the expand-and-contract pattern.

### Expand-and-contract pattern

The expand-and-contract pattern separates risky schema changes into safe stages.

Example: Rename `FullName` to `DisplayName`.

Unsafe one-step migration:

```csharp
migrationBuilder.DropColumn("FullName", "Customers");
migrationBuilder.AddColumn<string>("DisplayName", "Customers", nullable: false);
```

Safer staged approach:

Stage 1: Expand schema.

```csharp
migrationBuilder.AddColumn<string>(
    name: "DisplayName",
    table: "Customers",
    type: "nvarchar(200)",
    maxLength: 200,
    nullable: true);
```

Stage 2: Backfill data.

```sql
UPDATE Customers
SET DisplayName = FullName
WHERE DisplayName IS NULL;
```

Stage 3: Deploy code that writes both fields or writes the new field.

Stage 4: Make the new field required after data is complete.

```csharp
migrationBuilder.AlterColumn<string>(
    name: "DisplayName",
    table: "Customers",
    type: "nvarchar(200)",
    maxLength: 200,
    nullable: false,
    oldClrType: typeof(string),
    oldType: "nvarchar(200)",
    oldNullable: true);
```

Stage 5: Later, remove the old field after all code no longer depends on it.

This approach reduces deployment risk and makes rollback easier.

### Data migrations

A data migration changes existing data. It may be required when a schema change cannot work without transforming old rows.

Example:

```csharp
protected override void Up(MigrationBuilder migrationBuilder)
{
    migrationBuilder.AddColumn<string>(
        name: "NormalizedEmail",
        table: "Users",
        type: "nvarchar(320)",
        nullable: true);

    migrationBuilder.Sql("""
        UPDATE Users
        SET NormalizedEmail = UPPER(Email)
        WHERE Email IS NOT NULL AND NormalizedEmail IS NULL;
    """);
}
```

Data migrations need extra care because:

- They can be slow on large tables.
- They may lock rows or tables.
- They may fail due to unexpected data.
- They may need batching.
- They may need to be re-runnable.
- They may need monitoring.

For large systems, data backfills are often handled outside the schema migration using background jobs, ETL tasks, or manually reviewed scripts.

### Large table migration risks

Schema changes on large tables can be expensive.

Risky operations include:

- Adding a non-nullable column with a default to a large table.
- Creating a large index during peak traffic.
- Altering column type or length.
- Dropping or recreating indexes.
- Adding foreign keys that validate existing data.
- Updating every row in a table.
- Rebuilding a clustered index.
- Changing primary keys.

Safer approaches include:

- Add nullable columns first.
- Backfill in batches.
- Create indexes during maintenance windows when needed.
- Use online index operations if the database supports them.
- Test against production-like data volume.
- Measure execution time and lock behavior.
- Monitor blocking and deadlocks during deployment.

### Handling column renames

EF Core may not always infer a rename. It may generate a drop and add operation.

Unsafe generated code:

```csharp
migrationBuilder.DropColumn(
    name: "Name",
    table: "Products");

migrationBuilder.AddColumn<string>(
    name: "ProductName",
    table: "Products",
    type: "nvarchar(200)",
    nullable: false,
    defaultValue: "");
```

Safer rename:

```csharp
migrationBuilder.RenameColumn(
    name: "Name",
    table: "Products",
    newName: "ProductName");
```

If an index or constraint is tied to the old name, those may also need to be renamed or recreated carefully.

### Handling required columns

Adding a required column to a table with existing rows can fail unless a default is provided.

Example:

```csharp
migrationBuilder.AddColumn<DateTime>(
    name: "CreatedAtUtc",
    table: "Orders",
    type: "datetime2",
    nullable: false,
    defaultValueSql: "SYSUTCDATETIME()");
```

This may work, but it can still be risky on large tables depending on the database and operation.

A safer staged approach:

1. Add the column as nullable.
2. Backfill values.
3. Update application code to write the field.
4. Change the column to non-nullable.

### Seed data and migration noise

EF Core seed data using `HasData` can generate migration updates when seed values change.

Example issue:

```csharp
modelBuilder.Entity<Role>().HasData(new Role
{
    Id = 1,
    Name = "Admin",
    ConcurrencyStamp = Guid.NewGuid().ToString()
});
```

Because `Guid.NewGuid()` changes each time the model is built, EF Core may generate unnecessary `UpdateData` operations.

Better:

```csharp
modelBuilder.Entity<Role>().HasData(new Role
{
    Id = 1,
    Name = "Admin",
    ConcurrencyStamp = "2e69d8f4-7c2a-4d91-9c4f-000000000001"
});
```

Use stable seed values in migrations. Avoid dynamic values such as `DateTime.UtcNow` or `Guid.NewGuid()` in model seed data.

### Raw SQL in migrations

Sometimes EF Core migration operations are not enough. You can run raw SQL:

```csharp
migrationBuilder.Sql("""
    CREATE VIEW dbo.ActiveCustomers AS
    SELECT Id, Name, Email
    FROM dbo.Customers
    WHERE IsActive = 1;
""");
```

Raw SQL is useful for:

- Views.
- Stored procedures.
- Triggers.
- Complex data transformations.
- Provider-specific features.
- Index options not directly supported by EF Core.

But raw SQL should be reviewed carefully because:

- It may be provider-specific.
- It may not be validated at compile time.
- It can break idempotent scripts if not written carefully.
- It may require separate `Down` logic.
- Some statements need to run in a separate batch.

### Transaction behavior

EF Core generally attempts to apply migrations in transactions where possible. This helps avoid leaving the database partially migrated if something fails.

However, not all database operations can run inside a transaction on every provider.

Some operations may require transaction suppression:

```csharp
migrationBuilder.Sql(
    sql: "CREATE INDEX IX_Orders_CreatedAtUtc ON Orders (CreatedAtUtc);",
    suppressTransaction: true);
```

Use this carefully. Suppressing transactions can leave the database partially changed if a later operation fails.

Production scripts should be tested against the target database engine because transaction behavior differs by provider.

### Migration ordering and team workflow

In a team, multiple developers may create migrations at the same time.

Potential problems:

- Snapshot merge conflicts.
- Migration order conflicts.
- Duplicate schema changes.
- One branch modifies an entity that another branch also modifies.
- Migrations generated against stale model snapshots.
- A migration that works locally but fails after merging.

Good team habits:

- Keep migrations small and focused.
- Rebase or merge latest changes before generating migration.
- Resolve snapshot conflicts carefully.
- Build and run tests after merging migrations.
- Generate and inspect SQL after merge.
- Avoid editing old applied migrations.
- Create a new corrective migration instead of changing history after deployment.
- Test applying all migrations from a clean database.
- Test applying pending migrations from the previous released database.

### Editing migrations

It is acceptable to edit a newly generated migration before it is applied anywhere.

Common edits include:

- Replacing drop/add with rename.
- Adding custom SQL for data backfill.
- Fixing default values.
- Improving index definitions.
- Adjusting constraint names.
- Adding safer rollback logic.

However, once a migration has been applied to shared environments or production, editing it is dangerous. Other databases may already have recorded the migration in `__EFMigrationsHistory`.

In most cases, after a migration has been shared or applied, create a new migration to correct the issue.

### Removing migrations

If the latest migration has not been applied to a database, it can be removed:

```bash
dotnet ef migrations remove
```

This removes the last migration and reverts the snapshot.

If the migration has already been applied, first roll back the database to the previous migration, then remove it locally.

Example local rollback:

```bash
dotnet ef database update PreviousMigration
dotnet ef migrations remove
```

Do not remove or rewrite applied production migrations casually.

### Production migration checklist

Before production deployment, check:

- The migration builds successfully.
- The generated migration code was reviewed.
- The generated SQL script was reviewed.
- The migration was tested on a production-like database.
- The current production migration state is known.
- Backup or restore strategy is ready.
- Rollback script or recovery process is ready.
- Long-running operations were identified.
- Locking and blocking impact was considered.
- Data loss operations were explicitly approved.
- Backward compatibility was considered.
- Application deployment order is clear.
- Monitoring and alerts are ready.
- Database permissions are appropriate.
- Deployment owner and rollback owner are known.

### Deployment strategies

Common migration deployment strategies include:

| Strategy | Description | Best use |
|---|---|---|
| `dotnet ef database update` | EF tools apply migrations directly. | Local development and simple test environments. |
| Reviewed SQL script | Generate SQL, review it, then deploy it. | Most production systems. |
| Idempotent SQL script | Script applies only missing migrations. | Multi-environment or multi-tenant deployments. |
| Migration bundle | EF-generated executable applies migrations. | Controlled CI/CD deployment without source code on server. |
| Runtime `MigrateAsync` | App applies migrations at startup. | Development, demos, small controlled systems. |
| External migration tool | Tools such as DbUp, Flyway, Liquibase, or database projects manage scripts. | Teams requiring stronger DBA workflow or script-first control. |

### Least privilege for migrations

The application account should usually not have permission to change production schema.

A safer separation:

- Application user: read/write application data only.
- Migration/deployment user: schema modification permissions.
- DBA/admin user: emergency and administrative operations.

This supports least privilege. If the application is compromised, the attacker should not automatically be able to drop tables or alter schema.

### Migrations and CI/CD

A mature CI/CD workflow may include:

1. Build application.
2. Run unit tests.
3. Run integration tests.
4. Create a test database from scratch and apply all migrations.
5. Restore a copy of the previous release database and apply pending migrations.
6. Generate SQL script or migration bundle.
7. Publish migration artifact.
8. Require review or approval.
9. Apply migration in staging.
10. Run smoke tests.
11. Apply migration in production.
12. Monitor application and database health.

Example script generation in a build step:

```bash
dotnet ef migrations script PreviousReleaseMigration LatestMigration \
  --project src/MyApp.Infrastructure \
  --startup-project src/MyApp.Api \
  --output artifacts/database/migration.sql
```

Example idempotent script:

```bash
dotnet ef migrations script \
  --idempotent \
  --project src/MyApp.Infrastructure \
  --startup-project src/MyApp.Api \
  --output artifacts/database/migration-idempotent.sql
```

### Testing migrations

Migration tests can catch issues before production.

Useful tests include:

- Create a blank database and apply all migrations.
- Apply pending migrations to a database at the previous release version.
- Verify expected tables, columns, indexes, and constraints exist.
- Verify data backfills work.
- Verify old and new application versions can tolerate staged changes.
- Verify rollback script where rollback is part of the release plan.

Example integration test idea:

```csharp
[Fact]
public async Task AllMigrations_ShouldApplySuccessfully()
{
    var options = new DbContextOptionsBuilder<AppDbContext>()
        .UseSqlServer(TestConnectionString)
        .Options;

    await using var db = new AppDbContext(options);

    await db.Database.EnsureDeletedAsync();
    await db.Database.MigrateAsync();

    var pending = await db.Database.GetPendingMigrationsAsync();

    Assert.Empty(pending);
}
```

For realistic migration testing, prefer the same database provider used in production. EF Core InMemory is not appropriate for testing relational migration behavior.

### Common mistakes

Common mistakes include:

- Running unreviewed migrations in production.
- Applying migrations automatically at app startup without considering concurrency and permissions.
- Adding non-nullable columns to large tables without a safe plan.
- Letting EF Core convert a rename into drop-and-add.
- Dropping columns in the same release where old code may still need them.
- Using dynamic values in `HasData`.
- Ignoring the `Down` method.
- Assuming rollback can restore deleted data.
- Not testing against production-like data volume.
- Giving the application user schema modification permissions.
- Mixing `EnsureCreated` and migrations.
- Editing migrations after they have been applied to shared databases.
- Treating migrations as purely application code instead of deployment artifacts.

### Best practices

Good EF Core migration habits:

- Use descriptive migration names.
- Keep migrations small and focused.
- Review migration code.
- Review generated SQL.
- Test migrations before production.
- Generate SQL scripts for production review.
- Use idempotent scripts when target database state may vary.
- Use migration bundles when they fit your deployment model.
- Avoid runtime migrations for important production systems.
- Use expand-and-contract for breaking changes.
- Back up production before risky migrations.
- Plan rollback before deployment.
- Separate application permissions from migration permissions.
- Avoid destructive changes until a later release.
- Monitor database health during migration.
- Document migration assumptions in the release notes.
- Prefer provider-realistic integration tests.

#### What is an EF Core migration?

##### Expected Answer

An EF Core migration is a versioned schema change generated from changes in the EF Core model. It usually contains an `Up` method to apply the change and a `Down` method to reverse it. Migrations are stored as source code and committed to version control so the database schema can evolve consistently across environments.

When a developer changes entity classes or configuration, EF Core compares the current model with the previous migration snapshot and generates operations such as creating tables, adding columns, creating indexes, or changing constraints.

##### Key Points to Mention

- Migration represents an incremental database schema change.
- `Up` applies the change.
- `Down` reverses the change.
- Migrations should be checked into source control.
- EF Core generates migrations by comparing the current model to the model snapshot.

#### What command do you use to add and apply a migration?

##### Expected Answer

To add a migration with the .NET CLI:

```bash
dotnet ef migrations add AddCustomerEmail
```

To apply pending migrations to the configured database:

```bash
dotnet ef database update
```

In Visual Studio Package Manager Console, the equivalent commands are:

```powershell
Add-Migration AddCustomerEmail
Update-Database
```

In local development this is common. In production, it is usually safer to generate and review a SQL script instead of directly running `database update`.

##### Key Points to Mention

- `dotnet ef migrations add` creates a migration.
- `dotnet ef database update` applies migrations.
- Visual Studio has `Add-Migration` and `Update-Database`.
- Direct update is fine for local development.
- Production usually needs reviewed scripts or controlled deployment artifacts.
<!-- question:end:ef-core-migrations-beginner-q02 -->

<!-- question-level:beginner -->
#### What are the `Up` and `Down` methods in a migration?

##### Expected Answer

The `Up` method contains operations that apply the migration, such as adding a table, adding a column, or creating an index. The `Down` method contains operations that reverse the migration, such as dropping the added column or removing the created table.

For example, if `Up` adds an `Email` column, `Down` usually drops that column. The `Down` method is useful for rollback, but it does not automatically restore lost data if the `Up` method deleted or overwrote data.

##### Key Points to Mention

- `Up` moves the database forward.
- `Down` moves the database backward.
- `Down` helps rollback but cannot magically restore deleted data.
- Both methods should be reviewed.
<!-- question:end:ef-core-migrations-beginner-q03 -->

<!-- question-level:beginner -->
#### What is the EF Core model snapshot?

##### Expected Answer

The model snapshot is a C# file that represents EF Core's current understanding of the model after the latest migration. EF Core uses this snapshot when generating the next migration. It compares the current model against the snapshot to determine what changed.

The snapshot should be committed to source control. In a team environment, snapshot merge conflicts must be resolved carefully because an incorrect snapshot can cause future migrations to be generated incorrectly.

##### Key Points to Mention

- Snapshot stores the latest model state.
- EF Core compares the current model to the snapshot.
- It is used when generating future migrations.
- It should be committed to source control.
- Merge conflicts in the snapshot require careful review.
<!-- question:end:ef-core-migrations-beginner-q04 -->

<!-- question-level:beginner -->
#### What is the `__EFMigrationsHistory` table?

##### Expected Answer

`__EFMigrationsHistory` is a table EF Core uses to track which migrations have already been applied to a database. When EF Core applies migrations, it records each applied migration in this table. When generating or running idempotent scripts, EF Core can use this table to determine which migrations are missing.

Developers should not casually edit this table manually because it can cause EF Core to think a migration has or has not been applied incorrectly.

##### Key Points to Mention

- Tracks applied migrations.
- Used to determine pending migrations.
- Used by idempotent scripts.
- Should not be manually modified without a strong reason.

#### Why should production migrations usually use reviewed SQL scripts?

##### Expected Answer

Production migrations should usually use reviewed SQL scripts because database schema changes can cause data loss, locking, performance issues, or deployment failure. A SQL script shows the exact commands that will run against the database, so the team can inspect and test them before deployment.

Reviewed scripts allow developers, DBAs, and DevOps teams to detect dangerous operations such as dropping columns, rebuilding large indexes, adding non-nullable columns to large tables, or applying data transformations that could fail.

A typical production command is:

```bash
dotnet ef migrations script PreviousMigration NewMigration -o migration.sql
```

The generated SQL should then be reviewed, tested, archived, and deployed through a controlled process.

##### Key Points to Mention

- SQL review reduces production risk.
- Generated migration code may hide expensive SQL operations.
- Scripts can be tested and archived.
- DBAs can review locking, indexes, and data loss risk.
- Production deployment should be controlled and repeatable.
<!-- question:end:ef-core-migrations-intermediate-q01 -->

<!-- question:start:ef-core-migrations-intermediate-q02 -->
<!-- question-id:ef-core-migrations-intermediate-q02 -->
<!-- question-level:intermediate -->
#### What is an idempotent migration script and when would you use it?

##### Expected Answer

An idempotent migration script is a SQL script that checks which migrations have already been applied and only applies missing ones. EF Core uses the migration history table to do this.

Generate one with:

```bash
dotnet ef migrations script --idempotent -o migration.sql
```

Idempotent scripts are useful when deploying to multiple databases that may not all be at the same migration level, such as multi-tenant systems or environments that are not perfectly synchronized.

However, idempotent scripts should still be reviewed and tested. They are not a replacement for production safety practices.

##### Key Points to Mention

- Applies only missing migrations.
- Uses `__EFMigrationsHistory`.
- Useful for multiple databases or unknown current state.
- Still needs review and testing.
- Not always better than a targeted script when the source migration is known.
<!-- question:end:ef-core-migrations-intermediate-q02 -->

<!-- question:start:ef-core-migrations-intermediate-q03 -->
<!-- question-id:ef-core-migrations-intermediate-q03 -->
<!-- question-level:intermediate -->
#### Why can a column rename be dangerous in EF Core migrations?

##### Expected Answer

EF Core may not always know that a property rename means the database column should be renamed. It may generate a migration that drops the old column and adds a new column. This can cause data loss.

For example, changing `FullName` to `DisplayName` may generate:

```csharp
migrationBuilder.DropColumn("FullName", "Customers");
migrationBuilder.AddColumn<string>("DisplayName", "Customers", nullable: false);
```

A safer migration may use:

```csharp
migrationBuilder.RenameColumn(
    name: "FullName",
    table: "Customers",
    newName: "DisplayName");
```

This is why migrations must be reviewed before they are applied.

##### Key Points to Mention

- EF Core may detect rename as drop-and-add.
- Drop-and-add can lose data.
- Use `RenameColumn` when the intent is a rename.
- Review generated migrations carefully.
<!-- question:end:ef-core-migrations-intermediate-q03 -->

<!-- question:start:ef-core-migrations-intermediate-q04 -->
<!-- question-id:ef-core-migrations-intermediate-q04 -->
<!-- question-level:intermediate -->
#### Why is applying migrations at application startup risky in production?

##### Expected Answer

Applying migrations at application startup with `MigrateAsync` is risky because the application process directly changes the production schema. This requires elevated database permissions, makes SQL review harder, can fail during startup, and can create issues if multiple application instances start at the same time or if users access the database while migration is running.

Although newer EF Core versions add more protection against concurrent migration execution, startup migrations still do not solve all production concerns. They do not replace SQL review, rollback planning, least privilege, maintenance windows, and operational monitoring.

A safer production approach is to generate reviewed SQL scripts or migration bundles and run them as part of a controlled deployment process.

##### Key Points to Mention

- Requires app to have schema modification permissions.
- Can fail during startup.
- Can conflict with multiple app instances.
- SQL is not reviewed separately.
- Newer concurrency protections do not remove all operational risks.
- Prefer reviewed scripts or migration bundles for production.
<!-- question:end:ef-core-migrations-intermediate-q04 -->

<!-- question:start:ef-core-migrations-intermediate-q05 -->
<!-- question-id:ef-core-migrations-intermediate-q05 -->
<!-- question-level:intermediate -->
#### What is the difference between `EnsureCreated` and migrations?

##### Expected Answer

`EnsureCreated` creates a database schema directly from the current EF Core model. It bypasses the migrations system and does not create a proper migration history. It is mainly useful for tests, prototypes, or simple scenarios where the database does not need to evolve through migrations.

Migrations are used to incrementally evolve a schema over time. They track changes, generate migration files, and use the migration history table.

You should not mix `EnsureCreated` and migrations for the same relational database lifecycle because a database created by `EnsureCreated` does not have the migration metadata expected by `Migrate`.

##### Key Points to Mention

- `EnsureCreated` bypasses migrations.
- Migrations track schema history.
- `EnsureCreated` is not a production schema evolution strategy.
- Do not mix `EnsureCreated` and `Migrate` for the same database lifecycle.
<!-- question:end:ef-core-migrations-intermediate-q05 -->

<!-- question:start:ef-core-migrations-intermediate-q06 -->
<!-- question-id:ef-core-migrations-intermediate-q06 -->
<!-- question-level:intermediate -->
#### How would you safely add a required column to a large production table?

##### Expected Answer

A safe approach is usually staged:

1. Add the new column as nullable.
2. Deploy application code that can handle the nullable column.
3. Backfill existing rows in batches.
4. Update the application to always write the column.
5. Add a constraint or alter the column to be non-nullable after all rows have values.
6. Monitor the migration and application behavior.

Adding a non-nullable column with a default in one step may work on small tables, but on large tables it can cause long locks, expensive data updates, or deployment failure.

##### Key Points to Mention

- Avoid risky one-step schema changes on large tables.
- Add nullable first.
- Backfill in batches.
- Update application code safely.
- Enforce non-null only after data is ready.
- Consider locks, execution time, and rollback.
<!-- question:end:ef-core-migrations-intermediate-q06 -->

<!-- question-level:intermediate -->
#### What should you check when reviewing a generated migration?

##### Expected Answer

When reviewing a migration, check for data loss, unintended drops, incorrect renames, required columns without safe defaults, expensive index operations, foreign keys that may fail due to existing data, raw SQL risks, seed data noise, transaction behavior, and whether the `Down` method is meaningful.

You should also generate and inspect the SQL script because the actual SQL can reveal performance or locking risks not obvious from the C# migration code.

##### Key Points to Mention

- Look for dropped columns or tables.
- Check rename vs drop-and-add.
- Check nullable and default changes.
- Check indexes and foreign keys.
- Check raw SQL and data migrations.
- Review generated SQL, not only C# migration code.

#### How would you design a production-safe EF Core migration strategy?

##### Expected Answer

A production-safe strategy should treat migrations as deployment artifacts, not just developer commands.

A strong strategy includes:

1. Developers create small, focused migrations with descriptive names.
2. Migration code is reviewed in pull requests.
3. CI applies all migrations to a clean database.
4. CI applies pending migrations to a database representing the previous release.
5. CI generates SQL scripts or migration bundles.
6. SQL scripts are reviewed for destructive operations, locking, and performance.
7. The migration is tested in staging with production-like data volume.
8. Backups and rollback scripts are prepared.
9. The deployment process applies the migration with a controlled migration identity, not the normal application user.
10. Monitoring verifies success after deployment.

For breaking schema changes, use expand-and-contract so old and new application versions can coexist during rolling deployment.

##### Key Points to Mention

- Migrations should be reviewed and tested.
- SQL scripts or bundles should be deployment artifacts.
- Test from both clean database and previous release database.
- Use least privilege.
- Prepare rollback and backup.
- Use expand-and-contract for breaking changes.
- Monitor production after migration.
<!-- question:end:ef-core-migrations-advanced-q01 -->

<!-- question:start:ef-core-migrations-advanced-q02 -->
<!-- question-id:ef-core-migrations-advanced-q02 -->
<!-- question-level:advanced -->
#### Explain the expand-and-contract pattern for database migrations.

##### Expected Answer

Expand-and-contract is a safe deployment pattern for breaking schema changes. Instead of changing the schema in one risky step, the team separates the change into backward-compatible phases.

For example, to replace `FullName` with `DisplayName`:

1. Expand: add `DisplayName` while keeping `FullName`.
2. Deploy code that writes both columns or reads from both.
3. Backfill `DisplayName` from `FullName`.
4. Deploy code that uses `DisplayName`.
5. Contract: remove `FullName` in a later release after no code depends on it.

This pattern is useful for rolling deployments, blue/green deployments, distributed systems, and systems that cannot afford downtime.

##### Key Points to Mention

- Separates breaking changes into safe stages.
- Keeps old and new application versions compatible.
- Reduces rollback risk.
- Useful for zero-downtime or rolling deployments.
- Often needed for column rename, split, merge, or required-field changes.
<!-- question:end:ef-core-migrations-advanced-q02 -->

<!-- question-id:ef-core-migrations-advanced-q03 -->
<!-- question-level:advanced -->
#### How should rollback be handled for EF Core migrations?

##### Expected Answer

Rollback should be planned before deployment. EF Core can generate rollback scripts by scripting from a newer migration back to an older migration:

```bash
dotnet ef migrations script NewMigration PreviousMigration -o rollback.sql
```

However, rollback is not only a technical command. The team must know whether the migration is reversible, whether data would be lost, whether the old application version can run against the changed schema, whether a backup is needed, and how long recovery will take.

For destructive changes, a rollback script may recreate a column but cannot restore deleted data unless the data was preserved or restored from backup. For this reason, safe production systems often avoid destructive changes in the same release and use staged deployment patterns.

##### Key Points to Mention

- Generate rollback scripts when appropriate.
- Test rollback before production.
- Rollback cannot automatically restore lost data.
- Backups may be required.
- Old app version compatibility matters.
- Avoid destructive changes until a later release.
<!-- question:end:ef-core-migrations-advanced-q03 -->

#### What are migration bundles and when would you use them?

##### Expected Answer

Migration bundles are executable files generated from EF Core migrations. They can apply migrations without requiring the full project source code or EF CLI tools on the production server.

They are useful when a team wants an executable deployment artifact generated by CI/CD. A bundle can be versioned, approved, and executed as part of deployment.

However, bundles still need testing and operational controls. They should not be treated as a reason to skip migration review, rollback planning, permission design, or staging validation.

##### Key Points to Mention

- Bundle packages migrations into an executable.
- Useful as a CI/CD deployment artifact.
- Does not require source code on the target server.
- Still requires review, testing, and rollback planning.
- Can be an alternative to raw SQL scripts in some deployment models.

<!-- question-level:advanced -->
#### How do you handle EF Core migrations in a team where multiple developers create migrations?

##### Expected Answer

In a team, migrations should be small, focused, and frequently synchronized with the main branch. Developers should pull or rebase the latest changes before generating a migration. If multiple migrations are created in parallel, the team should carefully resolve snapshot conflicts, build the solution, inspect generated migrations, and test applying migrations from a clean database and from the previous release database.

If a migration has already been applied to shared environments, avoid editing it. Instead, add a new corrective migration. For migrations not yet shared or applied, it is usually acceptable to remove or edit the latest migration.

Some teams add CI checks to apply all migrations to a clean database and to verify pending migrations after model changes.

##### Key Points to Mention

- Snapshot conflicts must be resolved carefully.
- Keep migrations small and focused.
- Sync with main branch before generating migrations.
- Do not casually edit applied migrations.
- Test migrations in CI.
- Add corrective migrations when changes are already shared.

<!-- question-level:advanced -->
#### Why is least privilege important for EF Core migration deployment?

##### Expected Answer

Least privilege is important because schema modification permissions are powerful. If the normal application user can alter tables, drop columns, or create schema objects, then an application compromise can become a database schema compromise.

A safer design separates permissions:

- The application identity can read and write application data.
- The deployment or migration identity can apply schema changes.
- Administrative identities are reserved for exceptional operations.

This separation also makes deployments more auditable and reduces the blast radius of application bugs or security incidents.

##### Key Points to Mention

- Application user should usually not own schema changes.
- Migration identity can have elevated deployment-time permissions.
- Reduces security risk.
- Improves auditability.
- Supports production safety and compliance.
<!-- question:end:ef-core-migrations-advanced-q06 -->

<!-- question-level:advanced -->
#### How would you test EF Core migrations before production?

##### Expected Answer

Testing migrations should include more than checking that the application builds. A strong migration test strategy includes:

1. Apply all migrations to a clean database.
2. Apply pending migrations to a database at the previous production version.
3. Use the same database provider as production.
4. Test data migrations against realistic data.
5. Verify indexes, constraints, and expected schema objects.
6. Test application behavior after migration.
7. Test rollback if rollback is part of the deployment plan.
8. Measure execution time for large tables or expensive operations.

For serious production systems, testing against production-like data volume is important because a migration that succeeds on a small local database may lock or fail on a large production table.

##### Key Points to Mention

- Test clean database migration path.
- Test previous-release-to-current migration path.
- Use the real provider, not EF InMemory.
- Test data backfills.
- Test performance and locking risk.
- Test rollback where required.
<!-- question:end:ef-core-migrations-advanced-q07 -->

<!-- question-level:advanced -->
#### What are the risks of data migrations inside EF Core migrations?

##### Expected Answer

Data migrations can be risky because they modify existing production data. A simple update statement may scan a large table, hold locks, fail due to unexpected data, or take too long during deployment.

For small and predictable changes, data migration inside an EF Core migration may be acceptable. For large or business-critical data changes, it is often better to use a separate reviewed SQL script, background job, ETL process, or batched migration process.

A safe approach should consider batching, idempotency, monitoring, retry behavior, transaction scope, and rollback.

##### Key Points to Mention

- Data migrations can be slow and locking.
- Existing data may violate assumptions.
- Large updates should often be batched.
- Some data migrations should be separate from schema migrations.
- Need monitoring and rollback planning.
- Make scripts safe to re-run when possible.
